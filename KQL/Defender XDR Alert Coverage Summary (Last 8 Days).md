<h1>Defender XDR Alert Coverage Summary (Last 8 Days)</h1>

<p>
This KQL query provides a consolidated view of <strong>Microsoft Defender XDR alerts</strong>
over the <strong>last 8 days</strong>, grouped by detection solution, severity, and data source.
It is designed to support SOC operations, alert triage, and weekly security posture reporting.
</p>

<hr />

<h2>Overview</h2>

<p>
The query analyzes alerts from <strong>Advanced Hunting</strong> using the following tables:
</p>

<ul>
  <li><strong>AlertInfo</strong> – Core metadata for alerts generated by Defender workloads</li>
  <li><strong>AlertEvidence</strong> – Entities and artifacts associated with those alerts</li>
</ul>

<p>
It normalizes alert sources into consistent detection solution labels:
</p>

<ul>
  <li>MDE – Microsoft Defender for Endpoint</li>
  <li>MDO – Microsoft Defender for Office 365</li>
  <li>MDCA – Microsoft Defender for Cloud Apps</li>
  <li>MDI – Microsoft Defender for Identity</li>
</ul>

<hr />

<h2>What the Query Does</h2>

<h3>1. Normalizes Detection Sources</h3>
<p>
Different Defender workloads report alerts using varying <code>ServiceSource</code> values.
The query maps these values into unified labels (MDE, MDO, MDCA, MDI) to ensure
consistent reporting and analysis.
</p>

<h3>2. Builds Two Alert Datasets</h3>

<ul>
  <li>
    <strong>AlertInfoData</strong> – Summarized alert metadata (severity, title, creation time)
  </li>
  <li>
    <strong>AlertEvidenceData</strong> – Summarized alert-related entities and evidence
  </li>
</ul>

<p>
Both datasets are filtered to alerts generated within the last <strong>8 days</strong>.
</p>

<h3>3. Identifies Multi-signal Alerts</h3>

<p>
If an alert is associated with more than one Defender solution, it is classified as
<strong>Multi-signal</strong>. These alerts often represent correlated activity across
endpoint, identity, email, or cloud apps and typically indicate higher-confidence incidents.
</p>

<h3>4. Merges and Summarizes Results</h3>

<p>
The query merges both datasets and summarizes results by:
</p>

<ul>
  <li>Detection Solution</li>
  <li>Severity (Critical → Informational)</li>
  <li>Source Type (Alert vs AlertEvidence)</li>
</ul>

<p>
For each group, it provides:
</p>

<ul>
  <li>Total alert count</li>
  <li>Example Alert IDs (up to 5)</li>
  <li>Example Alert Titles (up to 3)</li>
</ul>

<hr />

<h2>Why This Is Valuable for Security Teams</h2>

<h3>Operational Visibility</h3>
<p>
Helps SOC teams quickly understand which Defender workloads are generating the most alerts
and where tuning or investigation effort should be focused.
</p>

<h3>Severity-driven Prioritization</h3>
<p>
Alerts are sorted using a custom severity ranking, allowing analysts to immediately focus
on <strong>Critical</strong> and <strong>High</strong> severity detections.
</p>

<h3>Multi-signal Awareness</h3>
<p>
Explicitly highlights alerts correlated across multiple Defender solutions, improving
detection confidence and reducing time-to-investigation.
</p>

<h3>Faster Investigations</h3>
<p>
Including example alert IDs and descriptions allows analysts to pivot directly into
incident investigation without additional hunting queries.
</p>

<h3>Foundation for Automation</h3>
<p>
The output can be reused for:
</p>

<ul>
  <li>Weekly SOC posture reports</li>
  <li>Detection coverage reviews</li>
  <li>Input into custom detection rules or automation workflows</li>
</ul>
